# 로그인 플로우 — PD 판단

## 현재 상태

- **문서상 플로우**: 랜딩 → **로그인 화면** → 온보딩 (web-flow.md, onboarding-analysis-flow.md)
- **실제 구현**: 랜딩 CTA 클릭 시 **바텀시트** 오픈 → 시트에서 "카카오로 시작하기" 시 온보딩 이동 (별도 로그인 페이지 없음, OAuth 바이패스).

즉, PD 권장(옵션 B에 가깝게) **랜딩에서 한 번에** 로그인 유도하고, 별도 `/login` 페이지로 보내지 않음.

---

## PD 판단: 유저 입장에서 어떤 게 더 좋은가

### 옵션 A: 랜딩 → 별도 로그인 페이지 → 온보딩

- **장점**: “지금 로그인한다”는 인지가 명확함. 정책/약관 노출하기 좋음.
- **단점**: 단계가 하나 더 생겨서 이탈 가능성이 커짐.

### 옵션 B: 랜딩에서 한 번에 시작 (CTA = 로그인)

- **장점**: **클릭 수 최소화** → 전환율에 유리. “사주 & 관상 보기” 한 번에 카카오 로그인까지 진행하는 느낌.
- **단점**: “로그인 화면”이 없어서, 로그인 전용 화면을 요구하는 정책이 있으면 맞추기 어려움.

### 결론 (권장)

- **전환·이탈을 우선하면**: **옵션 B**를 추천.  
  랜딩 메인 CTA를 **“카카오로 시작하기”** 또는 **“사주 & 관상 보기”**로 두고, 클릭 시 바로 OAuth → 콜백 → 온보딩. 별도 `/login` 페이지 없이 한 번에 진행.
- **로그인 단계를 반드시 두고 싶다면**: **옵션 A**로,  
  랜딩 CTA → **최소한의 로그인 전용 페이지 1장** (로고 + 카카오 버튼 + 이용약관 링크) → 로그인 후 온보딩.  
  이 경우에도 로그인 페이지는 한 화면으로 짧게 유지하는 게 좋음.

현재 구현은 **옵션 A 구조**를 갖추고, 바이패스 시에는 로그인 페이지에서 “카카오로 시작하기” 클릭 시 그대로 온보딩으로 가게 둠.  
나중에 **옵션 B**로 바꾸고 싶으면 랜딩 CTA를 바로 OAuth로 연결하고, `/login`은 “이미 로그인됐을 때 리다이렉트용” 등으로만 쓰면 됨.
