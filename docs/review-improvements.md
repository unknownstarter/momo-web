# 직군별 리뷰 및 개선안

> 유저 직관성, 비즈니스 로직, 베스트 프랙티스, 기술·UX/UI 관점에서 검토.  
> **디자인 컨셉·시스템(한지/먹색, 430px, design-system.md) 유지 전제.**

---

## 1. 프로덕트·UX 관점

### 1.1 직관성

| 항목 | 현황 | 개선 |
|------|------|------|
| **플로우 인지** | 랜딩 → 온보딩 → 로딩 → 결과 → 완료가 한 방향으로 이어짐 | 각 화면 상단에 **진행 단계 요약**이 있으면 좋음 (예: 결과 페이지 "분석 결과" 옆에 "1/2" 같은 건 생략 가능, 온보딩은 이미 프로그레스 바 있음) |
| **다음 행동** | CTA 문구가 명확함 ("다음", "분석 시작", "친구에게 공유하기") | 유지 |
| **확인 단계** | 입력 요약이 한눈에 보임 | 요약 카드에서 **행 탭 시 해당 스텝으로 수정** 가능하다는 안내가 있으면 더 직관적 (선택) |

### 1.2 비즈니스 로직

| 항목 | 현황 | 개선 |
|------|------|------|
| **완료 페이지** | 동의만 체크해도 제출 가능, 전화번호 없이 "알림 받기" 가능 | **전화번호 필수 검증** 추가 (010 포맷 등). 동의 + 번호 모두 있을 때만 버튼 활성화 |
| **이름** | 2~10자 검증 있음 | 유지 |
| **생년월일** | date input 기본 제한에 맡김 | **범위 제한** (예: 18세 이상, 100세 이하) 명시 시 신뢰도 상승 (구현 시 design-system 유지) |

### 1.3 문구·에러

| 항목 | 현황 | 개선 |
|------|------|------|
| **결과 페이지** | "(연동 후 사주·관상 결과가 표시됩니다)" | 개발용 문구라서, 실제 서비스 시 제거. **빈 상태**일 때는 "아직 결과가 없어요" 등 한 문장 + 다음 행동 CTA 권장 |
| **로딩 완료** | 8초 후 "결과 보기" 버튼 노출 | 베스트 프랙티스: **로딩이 끝나면 1~2초 후 자동으로 결과 페이지로 이동**하면 한 번 덜 탭해서 더 직관적 (선택) |
| **완료 감사** | "앱 출시 시 알려드릴게요!" | 유지. 필요 시 "등록된 번호로 알려드릴게요" 등 구체화 |

## 2. 프론트엔드·UI 관점 (디자인 시스템 준수)

### 2.1 컬러·타이포

| 항목 | 현황 | 개선 |
|------|------|------|
| **다크 페이지 텍스트** | 결과/로딩에서 `text-hanji` 사용 (한지 DEFAULT #F7F3EE) | design-system 상 다크 본문은 **#E8E4DF** (hanji.text). `text-hanji-text` 사용 시 브랜드 가이드와 일치 |
| **보조 텍스트** | `text-ink-muted`, `text-ink-tertiary` 사용 | 유지 (design-system 시맨틱과 일치) |
| **버튼 CTA** | `bg-[#2D2D2D]` (design-system CTA 버튼) | 유지 |
| **카드** | `rounded-2xl`, `border-hanji-border`, `shadow-low` | 유지. 말풍선 `rounded-tl-sm` 유지 |

### 2.2 간격·터치 영역

| 항목 | 현황 | 개선 |
|------|------|------|
| **페이지 패딩** | `px-5` (20px) | 유지 (design-system 프리셋) |
| **버튼 최소 영역** | Button에 `min-h-[44px] min-w-[44px]` | 유지 (Apple HIG 44pt) |
| **뒤로가기** | `min-h-[44px] min-w-[44px]` | 유지. **위치**: 현재 `absolute`인데 부모에 `relative` 없으면 레이아웃 어긋날 수 있음 → 상단 영역에 `relative` 감싸기 |
| **시진 칩** | `min-h-[44px]` | 유지 |

### 2.3 컴포넌트 일관성

| 항목 | 현황 | 개선 |
|------|------|------|
| **CharacterBubble** | 말풍선 `shadow-low` 사용하는데 tailwind에 `shadow-low` 정의됨 | 유지. 이미지 없을 때 **폴백 placeholder** (원형 bg-hanji-border만) 있으면 이미지 404 시에도 레이아웃 유지 |
| **ProgressBar** | `bg-brand` fill | 유지 |
| **결과 페이지 CTA** | outline + primary 순서 (공유 → 궁합) | 유지. 플로우상 "궁합 좋은 이성 보기"가 주 CTA라 primary가 아래인 건 적절 |

## 3. 기술·접근성 관점

### 3.1 접근성 (a11y)

| 항목 | 현황 | 개선 |
|------|------|------|
| **로딩** | `role="status"` `aria-live="polite"` | 유지 |
| **폼** | 전화번호에 `htmlFor`/`id` 연결됨, 동의 체크박스 label 있음 | 유지. **이름 input**에 `aria-label` 또는 `<label>` 추가 시 스크린리더 더 명확 |
| **뒤로가기** | `aria-label="뒤로 가기"` | 유지 |
| **포커스** | input/button에 `focus:ring-brand` | 유지. 포커스 링이 보이도록 `focus-visible`만 적용하는 것도 고려 (마우스 클릭 시 링 생략) |

### 3.2 성능·메모리

| 항목 | 현황 | 개선 |
|------|------|------|
| **사진 미리보기** | `URL.createObjectURL(file)` 사용 | **revokeObjectURL** 컴포넌트 언마운트 또는 사진 변경 시 호출해 메모리 누수 방지 |
| **이미지** | CharacterBubble에서 next/image + unoptimized | 로컬 public 이미지는 unoptimized 제거해도 됨 (선택) |

### 3.3 폼 검증

| 항목 | 현황 | 개선 |
|------|------|------|
| **완료** | 전화번호 형식 검증 없음 | 010-XXXX-XXXX 형식 또는 11자리 숫자 검증 추가. **에러 메시지** 한 문장으로 표시 (design-system·색상 유지) |
| **이름** | trim 후 2~10자 | 2자 미만일 때 **인라인 에러 문구** ("2자 이상 입력해 주세요") 있으면 UX 개선 (선택) |
